9:T331c,<p>この記事は <a href="https://qiita.com/advent-calendar/2019/react-native">React Native アドベントカレンダー</a>の 3 日目の記事です。</p>
<p>どうも、Naturalclar(<a href="https://twitter.com/natural_clar">@natural_clar</a>) です。
普段の業務では、<a href="https://expo.io/">Expo</a> ではなく <code>react-native init</code> で作成された、いわゆる素の React Native で開発しています。</p>
<p>先日の<a href="https://qiita.com/watanabe_yu/items/e59008eda79356d23918">ハムカツさんの記事</a>が今年の Expo に対する振り返りをやっていたので。
自分の方は素の React Native についての2019年の振り返りと、React Native の未来について語ろうと思います。</p>
<h2>2019/01</h2>
<h4>1/25 React Native v0.58 リリース</h4>
<p>v0.58 が今年最初のReact Native のリリースとなりました。
この辺から React Native でも TypeScript のサポートが増えてきた感触です。</p>
<p>今年は日本で <a href="https://typescript-jp.connpass.com/">TypeScript Meetup</a> が開催されたり、国内で TypeScript が爆発的に広まってきた一年でしたね。</p>
<p>今では<a href="https://github.com/react-native-community/react-native-template-typescript">react-native-typescript-template</a>を使って、コマンド一つで typescript 対応済みの react-native プロジェクトが作成できます。</p>
<pre><code>npx react-native init MyApp --template react-native-template-typescript
</code></pre>
<h2>2019/02</h2>
<h4>2/3 React Native Lean Core</h4>
<p>React Native のリポジトリに <a href="https://github.com/facebook/react-native/issues/23313">Lean Core</a> の Issue が立ちました。</p>
<p>Lean Core とは React Native の意向の一つで肥大化する React Native のリポジトリを軽くしようという試みです。
Checkbox, Async-Storage, WebView, Cli 等、必ずしも本体に含まれていなくても良い Package を React Native Community に移管して、React Native 本体から削減されていくことになります。</p>
<p>Umbrella 方式の <code>Good first issue</code> として立ち上がり、自分も参加コメントをしたのですが、まだ Native Component 周りの自信がなく、チキってしまったため、やれずじまいでした。</p>
<p>ここで Contribute して、分割された Package のメンテナーに名乗り出た人はもれなく React Native Community のメンバーになれて、Core Team の Discord にも招待されていたので、ここでやっていきを見せなかったことは今年一番の後悔した点です。</p>
<h2>2019/03</h2>
<h4>3/12 React Native v0.59 リリース</h4>
<p>v0.59 の目玉はなんと言っても Hooks ですね。
Hooks を導入した React 16.8 が React-native にも入ったので、多くの React Native 開発者が待ち焦がれていたのではないかと思います。</p>
<p>上述の Lean Core の Package も徐々にリリースされはじめました。
Lean Core に含まれている Package を React Native 本体から使っていた場合、Deprecation Warning が表示されていました。</p>
<p>また、React Native のアップグレードをするにあたって非常に便利なツールであった、<a href="https://github.com/pvinis">pvinis氏</a> の <a href="https://github.com/react-native-community/rn-diff-purge">rn-diff-purge</a> が react-native 公式でも勧められるようになりました。
rn-diff-purge は react native community に移管され、それを元に作成された GUIツールの <a href="https://react-native-community.github.io/upgrade-helper/">upgrade-helper</a> も実装されました。</p>
<p>React Native のアップグレードには毎回苦労されており、その度に rn-diff-purge にはお世話になったので、それが公式化されたことは個人的にとても喜ばしかったです。</p>
<h2>2019/07</h2>
<h4>7/3 React Native v0.60 リリース</h4>
<p>v0.60では、Native Module周りの仕様が大幅に変わりました。
Androidの方は すべての Package が Android X 対応され、Jetifier が React Native に自動的に組み込むようになりました。
iOS の方は Native Modules をすべて CocoaPods を使って組み込むようになりました。</p>
<p>今まで、<code>react-native link</code> を使ってつなぎ込んでいた Native Modules が Android では自動的に、iOS の方では <code>pod install</code> を実行するだけですべてが揃う様になりました。
このアップデートで Native Modules とのつなぎ込みはだいぶ楽になりました。</p>
<p>また、Android の方は Facebook が新規実装した JS のエンジンである <a href="https://github.com/facebook/hermes">Hermes</a> が試験的に導入されました。
Android の実行速度がこれにより格段に速くなりました。</p>
<p>前の Version で Deprecation Warning が表示されていた、 Lean Core で分割されたパッケージはここから React Native 本体からは外されました。</p>
<h4>7/11 react-navigation-hooks</h4>
<p>react-native の主要 Routing ライブラリの一つである <a href="https://reactnavigation.org/">react-navigation</a> が、Hooks ライブラリの <a href="https://github.com/react-navigation/hooks">react-navigation-hooks</a> をリリースしました。
これによって、今まで props で渡していた navigation を <code>useNavigation</code> を使って、Context内のどのComponentからも取得することができるようになりました。
より汎用的なnavigationが行えるようになり、とても便利になりましたね。</p>
<h4>7/11 react-redux v7.1.0</h4>
<p>同日に、<a href="https://github.com/reduxjs/react-redux/">react-redux</a> の7.1.0もリリースされました。</p>
<p>このアップデートでは <code>useSelector</code>や<code>useDispatch</code>が追加され、redux と繋ぎこむのに <code>connect</code> は不要の時代となりました。</p>
<p><code>redux-thunk</code> や <code>redux-saga</code> も hooks でリプレイス出来るようなった、まさに大Hooks時代の幕開けと言えたでしょう。</p>
<h3>2019/08</h3>
<h4>8/27 React Native v0.61 リリース</h4>
<p>v0.61にて、Fast Refreshが実装されました。
エミュレータ上で開発する上で、ファイルの変更の反映がかなり爆速になったので、Emulatorでそのまま実装するのも苦ではなくなりました。
自分は <a href="https://github.com/necolas/react-native-web">react-native-web</a> と <a href="https://storybook.js.org/">Storybook</a> を駆使して、Emulator を立ち上げずに react-native の実装を行っていたのですが、このバージョンからは Emulator 上でもサクサク実装出来るので、どっちも使うようになりました。</p>
<p>また、地味に Metro Bundler が <code>console.log</code> を出力するようになったので、<a href="https://github.com/jhen0409/react-native-debugger">react-native-debugger</a> などを立ち上げなくても debug が容易になりました。</p>
<p>この二つの変更と、v0.60 での Native Module の繋ぎこみがだいぶ楽になったおかげで、自分は正直Expoはもう使わなくても良いのではないかと思うくらい、素のReact Nativeでの開発が快適になったように感じます。</p>
<p>実際、Expo では共通の Expo サーバを使ったアプリのビルド待ちがあるため、Expo よりも React Native での開発のほうが効率的とさえ思います。</p>
<p>以上が、今年の React Native の大まかな振り返りとなります。</p>
<h2>React Nativeの今後について</h2>
<p>この一年で React Native は大幅に変更が加わり、開発体験もかなり良くなってきたと感じています。
では、これからの React Native はどうなっていくのでしょうか？</p>
<h4>次回のReact Native リリース</h4>
<p>次の React Native v0.62では、先日新調された <a href="https://github.com/facebook/react/tree/master/packages/react-devtools">react-devtools</a> が組み込めるようになり、 Nest が深い Component もだいぶ見やすくなる見込みです。</p>
<h4>React Navigation のアップデート</h4>
<p><a href="https://reactnavigation.org/docs/en/next/getting-started.html">React Navigation</a>は v5 が絶賛開発されています。
v5では今までの形と大幅に変わり、React-routerに近い形の使い方になるみたいです。
自分は今までのReact-navigationの形は悪くないと思っていたのですが、どういう風に変わっていくのか期待と不安が半々といったところです。</p>
<p>いまでも<a href="https://reactnavigation.org/docs/en/next/getting-started.html">react-navigation@next</a>で試せるので、興味があったらぜひ使ってみてください！</p>
<h4>Reason React Native の活動再開</h4>
<p><a href="https://reasonml.github.io/">Reason</a> と React Native の Bridge であった <code>bs-react-native</code> は、主な使用者であった Callstack が使用するのをやめてしまったため、しばらく動きが無い状態でした。
ですが、最近 <a href="https://reason-react-native.github.io/en/docs/">reason-react-native</a> として活動を復活されたので、個人的にはとても注目しています。</p>
<p>React Native v0.60が出た頃に活動が見受けられ、先日 v0.61 対応のアップデートが入りました。</p>
<p>Reason で書かれたコードはビルド時間も高速になり、開発体験がとても良い、今後の成長が期待出来る言語です。
Reason React Nativeはまだまだ人が少なくサポートも少ないので、自分が中の人になれるチャンスも大きくあると思っています。
今後自分でもライブラリのサポートやライブラリ自体に貢献していこうと思っています。</p>
<h4>モバイル以外にも広がっていく React Native</h4>
<p>React Native は基本的に iOS や Android のモバイルアプリを作成するものとして使われてきましたが、他にも色々な可能性を秘めています。</p>
<p>Chain React Confで発表された、<a href="https://github.com/microsoft/react-native">MicrosoftがForkしたReact Nativeのリポジトリ</a>も注目しています。
Microsoftでは<a href="https://github.com/microsoft/react-native-windows">react-native-windows</a>を使ってOffice 365等のデスクトップアプリを一部React Native を用いて開発しています。
この Microsoft が Fork した React Native のリポジトリでは、React Native を使って Mac OS のネイティブアプリを作れるように変更を加えていくとのこと。
Facebook, Microsoft, Appleの夢のコラボです。</p>
<p>AppleはAppleで、iPad用のアプリをMac OSのネイティブアプリに変換する<a href="https://developer.apple.com/mac-catalyst/">Mac Catalyst</a>を発表しました。
React Native で iPadのアプリを開発すれば、それもMac Catalystを使ってネイティブアプリに変換できる模様。
夢が広がりますね。</p>
<h4>更に未来へ</h4>
<p>React Native は今後、さらなる変更を控えています。</p>
<p>JavaScript を C++ に変換してさらなるマルチプラットフォームを狙う <code>JSI</code>、
Native Modules との繋ぎこみを加速し、さらなるネイティブの体験を提供していく <code>Turbo Modules</code>、
JS と Native Modules の繋ぎこみを自動化する <code>CodeGen</code>。
これらは <a href="https://github.com/react-native-community/discussions-and-proposals/issues/4">React Native Fabric</a> というプロジェクトで今後実装されていく予定のものです。</p>
<p>詳しくは先日、React Advanced にて React Native Community の Core Memberである Kelset が発表した React Native の未来にて語られています。
これからの React Native にもかなり期待できると個人的に思っています。</p>
<p>https://www.youtube.com/watch?v=7gm0owyO8HU</p>

<h2>最後に</h2>
<p>今年は React Native 本体や Cli などの周辺ツールに小さい Contribute をしてきましたが、
来年は本腰をいれて React Native Community の一員になるために活動していくつもりです。
React Native Communityにある何かしらのLibraryのMaintainerを目指していきます。</p>
